预备知识

Makefile文件中内容关系到整个工程的编译，内部定义了一系列的规则决定哪些文件需要先编译，哪些文件后编译，跟shell脚本类似。在学习makefile之前需要先简单了解下编译链接的原理。

程序的编译和链接

无论是C、C++语言都需要将源文件编译成中间文件，这个动作叫做编译(compile)，对于window下面这个中间文件就是.obj文件，在Unix下面就是.o文件。然后 再将.o文件链接成可执行文件，这个动作就是链接(link)。

gcc编译器

GNU CC(通常称为GCC)是GNU项目的编译器，它能够编译C、C++语言编写的程序。gcc是Linux开发的基础，gcc能够支持多种不同的C语言版本，同时提供Linux、Unix、Windows下的编译版本。gcc可以让跨平台编程更加容易，降低代码移植代价。

gcc编译C程序一般都是四个步骤：预处理、编译、汇编和链接。下面将介绍gcc常用的选项。

gcc常用选项

选项 含义
-c 只编译、汇编，不链接
-C 预处理时保留注释信息
-E 只进行预处理，不编译
-g 包含调试信息
-I（大写i） 指定include包含文件的搜索目录
-l 链接指定的库文件
-o [file1] [file2] 将文件 file2 编译成可执行文件 file1，如果不指定文件名则默认输出为a.out
-O[0,1,2,3] 优化程序，默认2级优化，0 不优化，优化和调试不兼容，所以不要同时使用-g 和-O 选项。
-S 只编译，不汇编

gcc高级选项

选项 详细
-v 详细输出编译过程中所采用的每一个选项
-C 预处理时保留注释信息
-w 忽略所有告警
-Wall 开启所有告警信息
-g 可生成能被 gdb 调试器所使用的调试信息
-ggdb 产生更多的调试信息

Makefile介绍

​ 在执行make命令时需要一个makefile文件，文件中的语法指定make执行时如何进行编译链接。make是一种控制编译或者重复编译软件的工具。make可以自动管理软件的编译内容、方式和时机，从而使程序员把更多的精力集中在编写代码上。

​ makefile是一个文本形式的脚本文件，其中包含一些规则告诉make编译哪些文件，怎么样编译以及在什么条件下编译。

​ Makefile核心内容：target中一个或多个目标文件依赖于dependency中的各个文件，生成规则在command中定义。一旦依赖文件有更新那么在make时command所定义的命令就会被执行。

make工作机制

​ 一般在目录下输入make命令即可执行当前目录下的makefile脚本：

  1. 执行make时会在当前目录中寻找makefile文件；
  2. 搜索到makefile脚本之后，会寻找最终需要生成的目标文件；
  3. 当目标文件不存在或者目标文件所依赖的.o文件修改有更新，则会继续向下执行下面的命令。整个make的依赖性，make会一层一层去找文件的依赖关系，直到最终编译；

​ 在找寻的过程中，如果出现错误，比如最后被依赖的文件找不到，那么make 就会直接退出，并报错，而对于所定义的命令的错误，或是编译不成功，make 根本不理。

make自动推导

make可以自动推导文件以及文件依赖关系后面的命令，这样就没有必要在每个.o文件后面写上类似的命令。只要make看到一个[.o]文件就会自动把[.c]文件加在依赖关系中。例如：make找到了main.o，那么main.c就会是main.o的依赖文件，并且gcc -c main.c也会被推导出来。

​ 既然提供了自动推导命令和文件功能，那么上面的.o和.h的依赖也能够收起来。

清空目标文件规则

​ 每个 Makefile 中都应该写一个清空目标文件（.o 和执行文件）的规则。

  • .PHONY 意思表示 clean 是一个“伪目标”；
  • 在 rm 命令前面加了一个小减号的意思就是，也许某些文件出现问题，但不要管，继续做后面的事；
  • clean 的规则不要放在文件的开头，不然，这就会变成 make 的默认目标；

Makefile文件之间引用

​ 在Makefile使用 include 关键字可以把别的Makefile包含进来，被包含的文件会原模原样的放在当前文件的包含位置。include的语法是：include 。注意在 include前面可以有一些空字符，但是绝不能是[Tab]键开始。include和filename之间可以用空格隔开。

如果文件都没有指定绝对路径或是相对路径的话，make 会在当前目录下首先寻找，如果当前目录下没有找到，那么，make 还会在下面的几个目录下找：

  • 若make执行时有“-I”或者“–include-dir”参数，那么make就会在这个参数所指定的目录下寻找；
  • 如果目录/include（一般是：/usr/local/bin 或/usr/include）存在的话，make 也会去找；

若未找到include所包含的文件：

​ 如果有文件没有找到的话，make 会生成一条警告信息，但不会马上出现致命错误。它会继续载入其它的文件，一旦完成 makefile 的读取，make 会再重试这些没有找到或者不能读取的文件。

​ 如果还是不行，make才会出现一条致命信息！如果不想理会这些无法读取的文件，让make操作继续往下执行，那么可以在include前面加一个减号“-“。

环境变量MAKEFILES

当前环境中定义了环境变量 MAKEFILES，那么，make 会把这个变量中的值做一个类似于 include 的动作。这个变量中的值是其它的 Makefile，用空格分隔。只是，它和 include不同的是，从这个环境变中引入的 Makefile 的“目标”不会起作用，如果环境变量中定义的文件发现错误，make 也会不理。

但是因为一旦变量被定义出来那么所有的makefile都会受到它的影响，故不建议使用。

Makefile综述

​ Makefile 里主要包含了五个东西：显式规则、隐晦规则、变量定义、文件指示和注释。

  • 显示规则：即显示的指出要生成的文件，文件的依赖文件，生成命令；
  • 隐晦规则：即之前的make的自动推导功能，隐晦的规则能简化书写makefile;
  • 变量定义：当 Makefile 被执行时，其中的变量都会被扩展到相应的引用位置上；
  • 文件指示：一是在某个Makefile中引用另外一个makefile；二是根据某些情况制定makefile中的有效部分，相当于C语言中的#if；三是定义一个多行命令。
  • 注释：Makefile中只有行注释，其注释用“#“字符，如果想在makefile中使用”#”只需要用转义字符，如”#“。

另外：在 Makefile 中的命令，必须要以[Tab]键开始。

make执行步骤

​ make工作时的执行步骤如下：

  1. 读入所有的makefile文件；
  2. 读入被include的其他makefile文件；
  3. 初始化文件中的变量；
  4. 推导隐晦规则，并分析所有规则；
  5. 为所有的目标文件创建依赖关系链；
  6. 根据依赖关系，决定哪些目标要重新生成；
  7. 执行生成命令；

make书写规则

​ 规则包含两个部分，一个是依赖关系，一个是生成目标的方法。

​ 在makefile中只应该有一个最终目标文件，其他的目标文件都是被带出来的，所以规则的顺序很重要。回看makefile工作机制中代码行注释。一般来说，make 会以 UNIX 的标准 Shell，也就是/bin/sh 来执行命令。

通配符

​ make中可以使用一些通配符来定义一系列类似的文件，**make支持三种通配符：“*”，“?”，“[…]“*。例如：“.c”表示所以后缀为 c 的文件，*如果我们的文件名中有通配符，如：“”，那么可以用转义字符“\”。

文件搜索

​ 一个工程中源件众多时，源文件被分配存放在不同目录下，当make需要去寻找文件的依赖关系时，可以在文件前加上路径名，但是最好的方法是把一个路径告知make，make自动寻找依赖文件。

​ Makefile文件中的特殊变量“**VPATH**”便是完成此项功能(如果没有指明该变量make只会在当前的目录中寻找依赖文件和目标文件）。

伪目标

​ clean操作用来清除所有中间文件，但是clean并不是一个目标文件，不会生成一个名为“clean”的文件。“伪目标”并不是一个文件，只是一个标签，所以在make时无法确定clean对应的命令是否要执行，那么只能通过显示地方式指明这个“目标”才能让其生效。

​ “伪目标”的取名不能和文件名重名，不然其就失去了“伪目标”的意义了！为了避免和文件重名的这种情况，我们可以使用一个特殊的标记“.PHONY”来显示地指明一个目标是“伪目标”，向 make 说明，不管是否有这个文件，这个目标就是“伪目标”

自动生成依赖性

​ 在makefile中依赖关系可能会包含一系列的头文件。当编译一个大型工程时就必须清楚知道哪些头文件，并且在增删文件时谨慎修改makefile文件，这样带来很多不便，此时就需要使用到C/C++编译器的一个功能：“-M”选项。

​ 如果你使用 GNU 的 C/C++编译器，得用“**-MM**”参数，不然，“-M”参数会把一些标准库的头文件也包含进来。

​ GNU 组织建议把编译器为每一个源文件的自动生成的依赖关系放到一个文件中。为每一个“name.c”的文件都生成一个“name.d”的 Makefile 文件，**[.d]文件中就存放对应[.c]文件的依赖关系**。

​ 于是可以写出[.c]文件和[.d]文件的依赖关系，并让 make 自动更新或生成[.d]文件，接着再包含在主makefile文件中，这样来自动化生成每个文件的依赖关系。下面给出一个模式规则用于产生[.d]文件。
